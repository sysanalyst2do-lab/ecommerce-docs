# Структура репозитория ecommerce-docs

## Общее описание

Репозиторий содержит техническую и бизнес-документацию для e-commerce системы. Документация организована по принципу разделения на уровни: бизнес-требования, системные требования, архитектурные решения и диаграммы.

## Описание разделов

### 📋 Требования (`docs/requirements/`)

#### Бизнес-требования (`business-requirements.md`)
Документ содержит бизнес-требования, организованные по доменам:
- **CAT** — Каталог и информация о товарах
- **CRT** — Корзина и предзаказ
- **CHK** — Оформление заказа
- **ORD** — Отслеживание, изменение и возвраты заказов
- **ACC** — Аккаунты и профили клиентов
- **CLI** — Управление клиентами и CRM
- **SUP** — Поддержка и обратная связь
- **INV** — Склад, запасы и выполнение заказов
- **MKT** — Маркетинг, акции и аналитика
- **X-BUS** — Сквозные бизнес-требования

Каждое требование имеет:
- Уникальный идентификатор (например, `BR-CAT-01`)
- Приоритет по методологии MoSCoW (M/S/C/W)
- Владельца требования
- Детальное описание

#### Системные требования (`system-requirements.md`)
Технические требования, вытекающие из бизнес-требований. Структура аналогична бизнес-требованиям, но с фокусом на техническую реализацию. Каждое системное требование содержит ссылки на связанные бизнес-требования для обеспечения трассируемости.

#### Нефункциональные требования (`non-functional-requirements.md`)
Требования к производительности, безопасности, доступности, масштабируемости и другим аспектам качества системы.

#### Матрица трассируемости (`traceability-matrix.md`)
Документ, обеспечивающий связь между бизнес-требованиями (BR), системными требованиями (SR) и архитектурными решениями (ADR).

### 🏗️ Архитектура (`docs/architecture/`)

#### Architecture Decision Records (`adr/`)
Документы, фиксирующие ключевые архитектурные решения:
- **ADR-0001**: Выбор архитектурного стиля (MVP монолит → платформа)
- **ADR-0002**: Добавление Kafka и интеграционных сервисов
- **ADR-0003**: Выделение доменных сервисов из монолита
- **ADR-0004**: NFR и архитектурные ограничения

Каждый ADR содержит:
- Дата и статус решения
- Контекст проблемы
- Принятое решение
- Последствия решения
- Связи с другими артефактами

#### C4 диаграммы (`c4/`)
Визуальное представление архитектуры системы в формате C4 Model:
- **C1 (Context)** — контекст системы и её взаимодействие с внешними акторами
- **C2 (Container)** — контейнеры (основные приложения и хранилища данных)

Три варианта архитектуры:
1. **MVP** — монолитный backend + React + PostgreSQL + Search DB
2. **Переходный вариант** — монолит + Kafka + выделенные интеграционные сервисы
3. **Платформа** — набор доменных сервисов + Kafka

Диаграммы хранятся в двух форматах:
- `.puml` — исходные файлы PlantUML (редактируются вручную)
- `.png` — сгенерированные изображения (обновляются автоматически через GitHub Actions)

## Принципы организации документации

1. **Трассируемость**: Все требования связаны между собой через матрицу трассируемости и прямые ссылки в документах.

2. **Иерархия**: Бизнес-требования → Системные требования → Архитектурные решения → Диаграммы.

3. **Версионирование**: ADR документируют эволюцию архитектурных решений с указанием дат и статусов.

4. **Визуализация**: Архитектура представлена в формате C4 Model для разных уровней детализации.

5. **Автоматизация**: Диаграммы генерируются автоматически из исходных PlantUML файлов.

## Рекомендации по использованию

### Для новых участников проекта:
1. Начните с `README.md` для общего понимания проекта
2. Изучите `docs/glossary.md` для понимания терминологии
3. Прочитайте `docs/requirements/business-requirements.md` для понимания бизнес-контекста
4. Ознакомьтесь с архитектурой через C4 диаграммы в `docs/architecture/c4/`
5. Изучите ключевые ADR для понимания принятых решений

### Для разработчиков:
1. Используйте системные требования (`system-requirements.md`) как спецификацию для реализации
2. Сверяйтесь с ADR при принятии архитектурных решений
3. Обновляйте диаграммы в формате `.puml` при изменении архитектуры

### Для архитекторов:
1. Ведите новые ADR при принятии значимых архитектурных решений
2. Обновляйте C4 диаграммы при изменении архитектуры
3. Поддерживайте актуальность матрицы трассируемости
