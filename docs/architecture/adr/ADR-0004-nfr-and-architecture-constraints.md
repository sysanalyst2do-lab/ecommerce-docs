## Контекст

Нефункциональные требования (производительность, доступность, безопасность, наблюдаемость, эксплуатация) накладывают ограничения на архитектуру, выбор технологий и организацию разработки.  
Нужно явно зафиксировать ключевые архитектурные решения, связанные с NFR, чтобы согласовать ожидания бизнес‑стейкхолдеров и команды разработки/эксплуатации.

## Решение

1. Производительность и масштабируемость
    - Использовать PostgreSQL как основную реляционную БД и отдельную Search DB (Elasticsearch/OpenSearch) для каталога.
    - Обеспечить возможность горизонтального масштабирования frontend и backend/сервисов (контейнеризация, оркестрация).
    - Ввести кэширование на уровне API/данных (при необходимости — последующим ADR).

2. Доступность и отказоустойчивость
    - Обеспечить раздельное масштабирование и изоляцию внешних интеграций от основной бизнес‑логики.
    - Использовать Kafka и асинхронный обмен для интеграций и части внутренних взаимодействий, чтобы уменьшать влияние отказов сторонних систем.
    - Определить стратегии retry, dead-letter очередей и деградации функционала при отказах (отдельные ADR в будущем).

3. Безопасность и соответствие
    - Использовать HTTPS для всех внешних и внутренних (межконтейнерных) HTTP‑вызовов в продакшене.
    - Вынести обработку платежей в отдельные сертифицированные сервисы (Payment Integration Service + внешние провайдеры), не хранить полные данные карт.
    - Реализовать ролевую модель и контроль доступа для административных функций и данных клиентов.
    - Хранить секреты и конфигурацию в защищённом хранилище (vault, секреты Kubernetes и т.п.).

4. Наблюдаемость
    - Внедрить централизованное логирование (например, ELK/EFK‑стек).
    - Настроить мониторинг (Prometheus/Grafana или аналог) для метрик сервисов, БД, Kafka.
    - Использовать корреляционные ID/trace ID для запросов и событий.

5. Эксплуатация и сопровождение
    - Внедрить CI/CD‑пайплайны для всех сервисов и монолита.
    - Определить стандарты версионирования API и событий в Kafka и поддерживать обратную совместимость при изменениях.
    - Документировать операционные процедуры (deploy/rollback, миграции схем, инцидент‑менеджмент).

## Последствия

### Положительные

- Явная связь между NFR и архитектурой, меньше «немых ожиданий».
- Проще принимать решения по масштабированию и техническому долгу.
- База для дальнейших более детальных ADR по специфическим NFR (performance tuning, security hardening).

### Отрицательные / Риски

- Возрастают требования к DevOps/инфраструктуре.
- Часть решений (Kafka, отдельные сервисы) усложняет архитектуру и требует опытной команды.

## Альтернативы

- Игнорировать NFR на старте и решать проблемы по мере возникновения — повышает риски серьёзных архитектурных переделок.
- Сразу вводить жёсткие Enterprise‑стандарты (SOA, ESB, сложные security‑контуры) — может замедлить MVP и увеличить стоимость без явной необходимости.
