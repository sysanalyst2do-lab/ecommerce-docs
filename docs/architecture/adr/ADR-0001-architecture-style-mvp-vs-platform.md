## Контекст

Нужно реализовать e-commerce систему, покрывающую каталог, корзину, оформление заказов, личный кабинет, маркетинг, поддержку и управление запасами.  
На старте важна скорость вывода MVP, но целевая картина — масштабируемая платформа с чёткими доменными границами и возможностью горизонтального масштабирования.  
Технологический стек: Python (backend), React (frontend), PostgreSQL (основное хранилище), нереляционная БД для поиска, Kafka — как внутренняя шина данных в целевом варианте.

## Решение

1. Для MVP принимается архитектура «монолит + отдельный фронтенд + отдельный поиск»:
    - React SPA как Web Frontend.
    - Python Backend (монолит: каталог, корзина, заказы, аккаунты, маркетинг, поддержка).
    - PostgreSQL как основная реляционная база данных.
    - Elasticsearch/OpenSearch как поисковый индекс каталога.
    - Интеграции с платёжным провайдером и провайдером уведомлений из монолита.

2. Целевая архитектура — набор доменных сервисов:
    - Catalog Service, Order Service, Customer Service, Inventory Service, Marketing Service, Support Service.
    - API Gateway/BFF для фронтенда.
    - Разделение данных по схемам/БД в PostgreSQL.
    - Отдельная Search DB для поиска.
    - Kafka для событийного обмена между сервисами и интеграциями.
    - Выделенные интеграционные сервисы (Payment, Notification, ERP/CRM).

3. Для перехода используется промежуточная архитектура (см. ADR-0002):
    - Монолит остаётся единственным доменным сервисом.
    - Добавляются Kafka и отдельные интеграционные сервисы.
    - Постепенно домены «вынимаются» из монолита в отдельные сервисы.

## Последствия

### Положительные

- Быстрый старт MVP за счёт монолита и минимального количества контейнеров.
- Постепенный путь к платформенной архитектуре без полного переписывания.
- Возможность включать Kafka и событийнyю интеграцию по мере роста нагрузки и числа интеграций.

### Отрицательные / Риски

- Монолит может усложниться и потребовать дополнительного рефакторинга перед выделением доменных сервисов.
- Потребуется управлять несколькими архитектурными состояниями (MVP, переходный вариант, целевая платформа).
- Нужна дисциплина по границам доменов уже на этапе монолита, чтобы облегчить дальнейший разнос.

## Альтернативы

- Сразу строить микросервисную архитектуру: отвергнуто для MVP из‑за высокой сложности и больших стартовых затрат.
- Оставаться в монолите до «последнего»: отвергнуто, так как не соответствует целевой платформенной стратегии и масштабируемости.
