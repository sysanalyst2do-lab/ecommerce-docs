# ADR-0002: Введение Kafka и выделенных интеграционных сервисов в архитектуру

## Контекст

MVP реализован как монолитный backend, но растёт количество интеграций:
- платёжные провайдеры,
- e-mail/SMS, мессенджеры,
- ERP/склад (остатки, заказы),
- CRM/Helpdesk.

Синхронные интеграции из монолита увеличивают связность, усложняют обработку ошибок и масштабирование. Необходимо ввести событийный обмен и изолировать интеграции.

## Решение

1. Добавить Kafka как внутренняя шина событий:
    - Монолит публикует доменные события (`order.*`, `product.*`, `customer.*`, `inventory.*`) в топики Kafka.
    - Интеграционные сервисы и будущие доменные сервисы подписываются на эти топики.

2. Выделить интеграционные сервисы из монолита:
    - Payment Integration Service:
        - принимает события/запросы на оплату из монолита,
        - вызывает платёжных провайдеров,
        - публикует события о статусах оплат.
    - Notification Service:
        - подписывается на события заказов, клиентов, маркетинга,
        - отправляет email/SMS/сообщения.
    - ERP Integration Service:
        - потребляет события по товарам, остаткам, заказам,
        - синхронизирует с ERP.
    - CRM Integration Service:
        - потребляет события по клиентам и обращениям,
        - синхронизирует с CRM/Helpdesk.

3. Монолит продолжает обслуживать весь домен (каталог, заказы, маркетинг, аккаунты), но:
    - не общается напрямую с внешними системами (кроме случаев, где это оправдано),
    - публикует события в Kafka, а интеграции выполняются внешними сервисами.

## Последствия

### Положительные

- Снижение связности монолита и упрощение его бизнес‑логики.
- Возможность масштабировать интеграционные сервисы независимо.
- Готовая событийная шина для дальнейшего выделения доменных сервисов.
- Улучшение устойчивости: внешние интеграции могут быть асинхронными.

### Отрицательные / Риски

- Усложнение инфраструктуры: нужен Kafka‑кластер и его сопровождение.
- Появление eventual consistency в некоторых сценариях (например, обновление остатков).
- Требуется мониторинг и логирование событий, управление схемами сообщений.

## Альтернативы

- Продолжать делать все интеграции из монолита синхронно: повышает риски деградации при проблемах внешних систем.
- Вводить point‑to‑point интеграционные сервисы без общей шины: усложняет дальнейшее развитие событийной архитектуры.
