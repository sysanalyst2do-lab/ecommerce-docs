# ADR-0003: Поэтапное выделение доменных сервисов из монолита

- Дата: 2026-03-05
- Статус: Draft
- Автор: <твоё имя>
- Связанные артефакты:
    - C4: `C2-mvp-ecommerce.puml`, `C2-transitional-ecommerce.puml`, `C2-platform-ecommerce.puml`
    - Требования:
        - BR: BR-ORD-01..05, BR-CAT-01..05, BR-INV-01..03, BR-MKT-01..04, BR-ACC-01..04, BR-CLI-01..03
        - SR: SR-ORD-*, SR-CAT-*, SR-INV-*, SR-MKT-*, SR-ACC-*, SR-CLI-*
        - NFR: NFR-PERF-03, NFR-AVAIL-01..02, NFR-OBS-01..03, NFR-OPS-01..03

## Контекст

После ввода Kafka и интеграционных сервисов монолит остаётся единственной реализацией доменов (каталог, заказы, клиенты, запасы, маркетинг, поддержка), что ограничивает масштабируемость и независимую эволюцию доменов.

Необходимо определить стратегию и порядок выделения доменных сервисов.

## Решение

Принят поэтапный план выделения доменных сервисов из монолита:

1. Этап 1 — Order Service:
    - Выделить подсистему «Заказы и корзина» в отдельный сервис.
    - Использовать Kafka для обмена событиями с монолитом (каталог, запасы, маркетинг).
    - Постепенно перевести UI/фронт на новый Order API.

2. Этап 2 — Catalog Service:
    - Выделить управление товарами и категориями.
    - Оставить Search DB за Catalog Service.
    - Публиковать события `product.*` в Kafka для Order, Marketing, Inventory.

3. Этап 3 — Inventory Service:
    - Выделить управление остатками и резервированием.
    - Перевести монолит/Order Service на использование Inventory API и событий.

4. Этап 4 — Marketing и Support:
    - Выделить Marketing Service (скидки, акции, персонализация).
    - Выделить Support Service или интеграцию с CRM/Helpdesk.

Общие принципы:
- Сначала выносить домены, которые создают наибольшую нагрузку и интеграционную сложность (Orders, Catalog, Inventory).
- Сохранять один источник правды для каждого домена (master system).
- На каждом этапе поддерживать совместимость с существующим фронтендом и интеграциями.

## Последствия

### Положительные

- Поэтапное снижение сложности монолита.
- Возможность масштабировать доменные сервисы по отдельности.
- Лучше соответствие доменной модели (bounded contexts).

### Отрицательные / Риски

- Временный период «гибридной» архитектуры, где часть логики в монолите, часть — в сервисах.
- Необходимость аккуратно управлять миграцией данных и маршрутизацией запросов.
- Увеличение требований к DevOps/наблюдаемости (логирование, трассировка, мониторинг).

## Альтернативы

- Big bang‑переписывание монолита на микросервисы: слишком рискованно и долго.
- Оставаться на монолите и только масштабировать его по горизонтали: не снимает проблемы эволюции доменов и независимой разработки.
